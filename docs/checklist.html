  1. Information About the Target

  1. Manually explore the website as an end-user and go through each function
  2. Manually explore the website as another access role (such as seller & editor or any other)
  3. Identify technologies, server and backend information
  1. Check HTTP header responses
  2. Tools: Whatweb, Builtwith, Wappalyzer & etc.
  1. Identify open ports and services
  1. Tools: Nmap, Sparta, Shodan
  1. Try to find public configuration files such as robots.txt, sitemap.xml, crossdomain.xml & other well-known files
  2. DNS information
  1. DNS Lookups, IP Space enumeration
  1. Google Dorking
  1. Try google dorks on the target to identify sensitive data such as "Site: thecomputerjoker.com inurl:admin.php"
  1. Github reconnaissance
  1. Try searching about the target on GitHub, you may find sensitive data such as source code, API, credentials & etc.
  1. Find comments on source code
  1. Burp engagement tools
  1. Directory enumeration
  1. Tools: Dirbuster
  1. Get URLs about the target
  1. Tools: waybackurls
  1. Get ASN and IP Ranges
  1. asnlookup
  1. Get JS files & analysis
  1. Tools: subjs, linkfinder, JSParser
  1. Find Hardcoded APIs and other secrets
  1. Tools: secretfinder
  1. Parameter Fuzzing
  1. Tools: FFuF
  1. Port Scanning
  1. Tools: Nmap
  1. Enumerate Sub-domains
  1. Tools: sublister
  1. Sub-domain bruteforce
  1. Tools: subbrute
  1. Sub-domain takeover
  1. Tools: SubOver
  1. Check for cloud assets
  1. Tools: Cloudenum, cloudscrapper & cloudlist
  1. Take Screenshots of the links
  1. Tools: webscreenshot
  1. Run template-based scanning
  1. Tools: nuclei
  1. Check for backup files
  1. Tools: bfac
  1. Broken Link Hijacking
  1. Tools: blc
  1. Test CORS configuration
  1. Tools: CORScanner
  1. Automated Scanning
  1. Tools: Burp Suite
  1. Dark-web discovery
  1. Tools: intelx
  1. Discovery withOpen-Souce tools
  1. Tools: OSINT
  1. Test SSL details
  1. Tools: ssltest
  1. Check DMARC and SPF records
  1. Tools: mxtoolbox

  1. Registration checks

  1. Duplicate/overwrite registration
  2. Try with upper/lower case, +1@, dots in the name, @ to %40, etc.
  3. Weak password policy
  4. Check for username duplication
  5. Fail to validate the email verification
  6. Registration with a disposable email
  7. Put only spaces in the password
  8. Registration on the HTTP
  9. Check for insufficient email verification process
  10. Check for an upper cap for password
  1. password with 100+ characters may lead to application DOS
  1. Rate limiting on register page

  1. Authentication checks

  1. Username Enumeration
  1. Put any username and check if the application responds with "username already taken"
  1. Remember me / Autocomplete function in the browser
  2. SQL injection to bypass authentication
  1. uname='1'or'1'='1' and pass='1'or'1'='1';
  1. Lack of password confirmation on Email change
  2. Lack of password confirmation on change password
  3. Lack of password confirmation on Manage 2FA
  4. Lack of password confirmation on Delete Account
  5. Login over HTTP (cleartext submission of password)
  6. No account lock mechanism (No rate limiting on login - brute-force)
  1. Try brute-force by changing user-agent
  2. Try brute-force with a null byte
  1. Test Oauth configuration
  2. Test 2FA configuration
  1. Response maniplulation
  2. Status code change
  3. 2FA code does not change for a certain time
  4. 2FA code reusability
  5. 2FA code leakage in response
  6. With NULL or 000000
  7. 2FA brute-force protection

  1. Session management

  1. Test token format/predictability
  2. Identify session cookie among all the cookies
  3. Try to decode cookies
  1. Tools: crack station, burp suite decoder
  1. Check cookie expiration
  2. HTTPonly & Secure flag on the cookie
  3. Session fixation
  1. Session cookie gets changed before and after login
  1. Use the same cookie from different machine/IP
  2. Check for concurrent login from different machines/IP
  3. Reuse of cookie after logout
  4. Session gets terminated on logout
  1. Logout and press the back button to check if the session gets terminated or not
  1. Failed to invalidate session on password change
  2. Failed to invalidate session on email change
  3. Failed to invalidate session on the reset password
  4. Failed to invalidate session on enable/disable 2FA

  1. Profile checks

  1. Check for data of another user by changing user_id or profile_id
  2. Check for CSRF on important functions such as changing password, profile update, contact update, etc.
  3. Check the email change feature, after updating the new_email, whether the existing_email gets notified of any change that can be made from the existing_email instance.
  4. Check for EXIF data on images
  5. CSV injection via file download
  6. Check metadata of downloadable files
  7. Account deletion feature
  1. Delete the account and go for forgot password and check the response from the application
  1. Bruteforce on the username & other profile parameters
  2. Parameter pollution by putting two values in the same field
  3. Check if the application asks for re-authentication on sensitive operations such as payment

  1. Forget Password checks

  1. Terminate session on logout and password reset
  2. The uniqueness of reset password link/code
  3. Rest link expiration time
  4. Rest link expires after one-time use
  5. Find any user_id in the reset link and try to change them
  6. Request multiple reset password links and use the oldest
  7. Check if the reset password links have a sequential pattern
  8. Use user@burpcollab.net and analyze the callback
  9. Password reset token leakage in the referrer
  10. Check for weak password policy
  11.  If the reset link has any time/date then tamper it and try to load
  12. Any security questions asked? How many? Check for brute-force on the same
  13. Invalidate the current instance after resetting the password
  14. Password reset token on HTTP
  15. No rate-limiting on forget password page leads to email flooding
  16. 2FA gets disabled on password reset
  17. Try to decode reset password token
  18. Response manipulation
  19. Add one more email parameter in the request and values
  20. Understand the token pattern(birthdate,username,email & etc)

  1. Input Validation

  1. Fuzz all the request parameter
  2. Host Header injection
  1. Supply arbitrary host header
  2. Add duplicate header
  3. Add line wrapping
  1. SQL injection
  1. Find parameters vulnerable to SQL (mostly IDs, User-supplied input which directly parsed into a database), you can use sqlmap for it.
  2. ID-based(error-based) SQL injection
  3. Blind SQL injection
  4. Time delay SQL injection
  5. Conditional delay - SQL injection
  6.  Bypass WAF
  1. Use encoding
  2. Use tamper scripts
  3. Insert NULL
  4. Change the case(upper/lower)
  1. Cross-Site Scripting
  1. Try automated scanner for XSS
  1. Tools: dalfox, quickxss
  1. If the SCRIPT tag is not working, try H1 tags
  2. Check for vulnerable inputs
  1. Try adding script and check in the response(source code), if the special characters are not encoded, it may be vulnerable to XSS
  1. XSS payloads
  2. XSS filter evasion
  3. XSS firewall bypass
  1. OS command injection
  2. Path traversal
  3. LFI & RFI
  4. SOAP Injection
  5. LDAP Injection
  6. SSI Injection
  7. XPATH Injection
  8. XXE
  1. Change the content type text/XML, add XML data and check the response
  2. Blind XXE with out-of-band interaction
  1. NoSQL Injection
  2. Try to discover hidden parameter
  1. Tools: arjun or parameth

  1. Error Handling

  1. Access custom pages like /whaterer.php on target
  2. Add “[]”, “[[” in cookie parameter and check for response
  3. Generate error by putting random special character at the end of URL
  4. User burp intruder and fuzz the input to generate errors
  5. Try different HTTP methods like DEBUG, PATCH or wrong (FAKE)

  1. CAPTCHA checks

  1. Reusable captcha
  2. Load the page with adblocker
  3. Content-type conversion
  4. Bypass with OCR tool
  5. Try changing the HTTP method
  6. Remove captcha parameter from the request or GUI(inspect element)

  1. Miscellaneous

  1. Access role controls
  1. IDOR (Works by changing the ID of other user/role)
  2. Check if one user role can access another user role’s data
  3. Parameter tampering to access high privilege role
  1. Unauthorized download
  1. Example: Invoice pdf in your account, get that link of the document and try to load it in incognito mode
  1. Username enumeration on the Unsubscribe page
  2. Change POST to GET insensitive requests
  3. Check for internal IP in the code
  4. Check for weak cipher used in SSL
  5. Denial of Services(DOS)
  1. DOS on the web application
  2. Application DOS by entering 200+ characters in all input parameters
  3. Pixel flood (use image with huge pixels)
  1. Cross-Origin Resource Sharing
  1. Parse origin headers
  1. Server-Side Request Forgery
  2. Open redirection
  3. Contact us form
  1. No rate limiting leads to email flooding
  2. File uploading on the server
  3. Blind XSS
  1. Tools: XSShunter
  1. Parameter tampering(payment value, product_id, gift_voucher, etc)
  2. File Upload
  1. Upload large files may lead to DOS
  2. Metadata leakage
  3. ImageMagick attack
  4. Pixel Flood
  5. Upload file with NULL bytes
  6. Upload file by changing the content-type
  1. Word-press checks
  1. XMLrpc exposed, may lead to DDOS
  2. Directory enumeration
  3. Brute-force on login
  4. Wp-config and wp-includes information disclosure
  5. Logfile could be disclosed
  1. HTTP Security Headers
  1. X-XSS protection
  2. Strict-Transport-Security
  3. X-frame options
  4. Content-security-policy
  5. X-content-type-options
  6. Referer-policy
  7. Cache-control
  8. Expires
